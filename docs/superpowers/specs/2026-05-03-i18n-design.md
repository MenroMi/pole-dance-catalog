# i18n â€” Design Spec

_2026-05-03_

## Overview

Add Polish + English support to the full application â€” UI strings, dynamic DB content, and email notifications. Polish is the default language. Language preference is detected from the browser's `Accept-Language` header on first visit, then persisted in a cookie. Users can switch language at any time via a globe icon dropdown in the Header.

**Library:** `next-intl` (built for Next.js App Router; handles both Server and Client Components natively).

**Languages:** `pl` (default), `en`.

---

## 1. Routing

### URL structure

All routes are prefixed with the locale segment:

```
/pl/catalog
/en/catalog
/pl/moves/[id]
/en/moves/[id]
/pl/login
/en/login
/pl/profile
/en/profile
/pl/admin
/en/admin
```

The root `/` redirects to `/${detectedLocale}/` via middleware.

`/api/*` routes have no locale prefix â€” they return JSON and do not need localisation.

### Locale config

```ts
// src/i18n/routing.ts
export const locales = ['pl', 'en'] as const;
export type Locale = (typeof locales)[number];
export const defaultLocale: Locale = 'pl';
```

## 2. Middleware

`src/middleware.ts` replaces the current `src/proxy.ts`. It handles two concerns in sequence:

1. **Locale routing** â€” next-intl's `createMiddleware` detects locale from:
   - Cookie `NEXT_LOCALE` (user's stored preference)
   - `Accept-Language` request header (browser preference)
   - Fallback: `pl`

   Redirects `/` â†’ `/pl/` (or `/en/`). Validates locale in URL â€” invalid locales (e.g. `/de/catalog`) are redirected to the default locale.

2. **Auth guards** â€” after locale is resolved, protected routes (`/[locale]/profile/*`, `/[locale]/admin`) redirect to `/[locale]/login` when unauthenticated.

```ts
// src/middleware.ts (simplified)
import createMiddleware from 'next-intl/middleware';
import { routing } from './i18n/routing';

const intlMiddleware = createMiddleware(routing);

export default async function middleware(req: NextRequest) {
  const intlResponse = intlMiddleware(req);
  if (intlResponse) return intlResponse;
  return authGuard(req);
}

export const config = {
  matcher: ['/((?!api|_next|.*\\..*).*)'],
};
```

Cookie `NEXT_LOCALE` is set automatically by next-intl middleware when it detects the locale from the URL â€” no manual cookie handling needed.

## 3. File Structure

### App directory

```
src/app/
  layout.tsx              â€” minimal root: <html><body> only, no locale-specific content
  global-error.tsx        â€” stays at root; hardcoded strings (no locale context on critical crash)
  api/                    â€” unchanged, no locale prefix
  [locale]/
    layout.tsx            â€” sets lang attr, mounts NextIntlClientProvider, loads fonts
    page.tsx              â€” landing page (moved from src/app/page.tsx)
    not-found.tsx         â€” locale-aware 404 (replaces src/app/not-found.tsx)
    (auth)/               â€” internal structure unchanged
    (main)/               â€” internal structure unchanged
    admin/
      page.tsx
```

**Root `src/app/not-found.tsx` is deleted.** Middleware always redirects unknown paths to a valid locale prefix before Next.js renders anything, so `[locale]/not-found.tsx` handles all 404 cases.

> **Note (tech debt):** There is a theoretical edge case where Next.js triggers a 404 before middleware runs (e.g. certain static file requests). In practice this does not occur with the current matcher config, but worth monitoring post-launch.

**`global-error.tsx` uses hardcoded strings.** It fires when the root layout or `NextIntlClientProvider` itself crashes â€” translations are unavailable in this scenario. Using hardcoded strings is intentional and standard practice.

> **Note (tech debt â€” suggestion):** If translated error messages in `global-error.tsx` are desired in the future, it is possible to initialise next-intl manually without a Provider by loading the JSON file directly. Low priority.

### i18n config

```
src/i18n/
  routing.ts      â€” locales, defaultLocale (imported by middleware and components)
  request.ts      â€” next-intl server-side config (getRequestConfig)
  messages/
    pl.json       â€” Polish (default)
    en.json       â€” English
```

### generateStaticParams

`[locale]/layout.tsx` exports:

```ts
export function generateStaticParams() {
  return locales.map((locale) => ({ locale }));
}
```

Next.js automatically combines this with `generateStaticParams` in nested dynamic segments (e.g. `moves/[id]/page.tsx`) to produce all locale Ă— id combinations. No changes needed to existing `generateStaticParams` implementations.

### TypeScript type safety

`pl.json` is the source of truth for the message shape. `src/i18n/types.ts` augments `IntlMessages`:

```ts
import pl from './messages/pl.json';
type Messages = typeof pl;
declare global {
  interface IntlMessages extends Messages {}
}
```

TypeScript will error if `en.json` is missing any key present in `pl.json`. No separate test needed.

## 5. Database Schema

### Translatable fields

All translatable fields are split into `_pl` and `_en` variants. **Both are required** â€” the Admin UI enforces that both language versions are filled simultaneously. There is no runtime fallback between languages; missing content is a data error, not a handled case.

**`Move` model changes:**

| Old field             | New fields                                         |
| --------------------- | -------------------------------------------------- |
| `title String`        | `title_pl String`, `title_en String`               |
| `description String?` | `description_pl String?`, `description_en String?` |
| `stepsData Json`      | `stepsData_pl Json`, `stepsData_en Json`           |
| `gripType String?`    | `gripType_pl String?`, `gripType_en String?`       |
| `entry String?`       | `entry_pl String?`, `entry_en String?`             |
| `coachNote String?`   | `coachNote_pl String?`, `coachNote_en String?`     |

`coachNoteAuthor String?` â€” unchanged (person's name, not translated).

Non-translatable fields unchanged: `difficulty`, `category`, `poleTypes`, `youtubeUrl`, `imageUrl`, `stepsData` structure (only text content inside steps is translated), `createdAt`, `updatedAt`.

**`Tag` model changes:**

| Old field             | New fields                                         |
| --------------------- | -------------------------------------------------- |
| `name String @unique` | `name_pl String @unique`, `name_en String @unique` |

**`VerificationToken` model â€” add `locale` field:**

```prisma
model VerificationToken {
  // existing fields...
  locale String @default("pl")
}
```

**`PasswordResetToken` model â€” add `locale` field:**

```prisma
model PasswordResetToken {
  // existing fields...
  locale String @default("pl")
}
```

### Migration strategy

No production data exists. Migration is:

```bash
prisma migrate reset   # drops and recreates schema
# update seed scripts to include both _pl and _en content for all moves and tags
prisma db seed
```

All seed scripts must provide both `_pl` and `_en` values for every move and tag from the start.

### Localisation helper

```ts
// src/shared/lib/localize.ts
export function localizeMove(move: RawMove, locale: Locale) {
  const pick = <T>(pl: T, en: T) => (locale === 'pl' ? pl : en);
  return {
    ...move,
    title: pick(move.title_pl, move.title_en),
    description: pick(move.description_pl, move.description_en),
    stepsData: pick(move.stepsData_pl, move.stepsData_en),
    gripType: pick(move.gripType_pl, move.gripType_en),
    entry: pick(move.entry_pl, move.entry_en),
    coachNote: pick(move.coachNote_pl, move.coachNote_en),
  };
}

export function localizeTag(tag: RawTag, locale: Locale) {
  return { ...tag, name: locale === 'pl' ? tag.name_pl : tag.name_en };
}
```

All actions that return `Move` or `Tag` data accept `locale: Locale` and apply the localise helper before returning.

## 6. Language Switcher

### Component

A `'use client'` component rendered inside the existing `Header`. Uses `@radix-ui/react-dropdown-menu` (already installed) â€” same pattern as `UserMenu`.

**Trigger:** globe icon (Lucide `Globe` or inline SVG).

**Dropdown items:**

- `Polski` â€” active state (check mark) when `locale === 'pl'`
- `English` â€” active state when `locale === 'en'`

**Navigation:**

```tsx
'use client';
import { useRouter, usePathname } from 'next-intl/navigation';
import { useLocale } from 'next-intl';

export function LocaleSwitcher() {
  const locale = useLocale();
  const router = useRouter();
  const pathname = usePathname(); // path WITHOUT locale prefix

  function switchLocale(next: Locale) {
    router.replace(pathname, { locale: next });
  }
  // ...
}
```

`usePathname` from `next-intl/navigation` returns the path without the locale prefix (e.g. `/catalog`, not `/pl/catalog`). `router.replace` from `next-intl/navigation` handles prepending the new locale. The next-intl middleware sets cookie `NEXT_LOCALE` automatically on the next request.

## 7. Emails

### Locale storage

Locale is stored in the token DB record at the time the email is sent. This ensures the post-action redirect uses the correct locale regardless of the device or browser used to open the email link.

`VerificationToken.locale` and `PasswordResetToken.locale` are set from `getLocale()` in the Server Action that triggers the email.

### Verification flow

1. `signupAction` / `resendVerificationAction` calls `getLocale()`, creates `VerificationToken` with `locale` field, calls `sendVerificationEmail(email, token, locale)`
2. `sendVerificationEmail` uses `getTranslations({ locale, namespace: 'emails.verification' })` for subject and body
3. Verification URL in email: `/api/auth/verify?token=abc` (no locale in URL)
4. Route handler `/api/auth/verify` reads `verificationToken.locale` from DB after finding the token â†’ redirects to `/${token.locale}/login?verified=true`

### Password reset flow

Same pattern: `forgotPasswordAction` stores locale in `PasswordResetToken`. Reset URL in email: `/api/auth/reset-password?token=abc`. After successful reset, route (or action) redirects to `/${token.locale}/login?reset=true`.

## 8. Testing

### Unit tests

**`localizeMove` / `localizeTag` (pure functions):**

- Returns `_pl` fields when `locale === 'pl'`
- Returns `_en` fields when `locale === 'en'`

**`LocaleSwitcher` (RTL):**

- Renders globe icon trigger
- Opens dropdown with `Polski` and `English` items
- Clicking `English` calls `router.replace` with `{ locale: 'en' }`
- Active locale item is visually distinct

**Email functions (mocked `getTranslations`):**

- `sendVerificationEmail(..., 'pl')` sends with Polish subject
- `sendVerificationEmail(..., 'en')` sends with English subject

### Updated tests

**Middleware tests** (extend current `proxy.test.ts` pattern):

- `/` with no cookie + `Accept-Language: pl` â†’ redirects to `/pl/`
- `/` with cookie `NEXT_LOCALE=en` â†’ redirects to `/en/`
- `/pl/profile` unauthenticated â†’ redirects to `/pl/login`
- `/en/profile` unauthenticated â†’ redirects to `/en/login`

**Route handler `/api/auth/verify`:**

- Expired token â†’ redirects to `/pl/verify-email?error=expired` (using `token.locale`)
- Valid token â†’ redirects to `/pl/login?verified=true` (using `token.locale`)

### TypeScript as test

`src/i18n/types.ts` augmentation means TypeScript errors at compile time if `en.json` is missing any key from `pl.json`. This replaces a dedicated JSON completeness test.
